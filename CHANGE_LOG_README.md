**CHANGE LOG 規範（簡要）**

- **位置**: 所有變更日誌置於專案根目錄下的 `log` 資料夾。
- **檔名格式**: `YYYY-MM-DD_HHMMSS_log.md`，例如 `2025-11-18_120000_log.md`，使用本地時間作為時間戳記。
- **每次變更**: 建議為每次由維護者或自動化工具所做的「實際檔案修改」建立一個獨立的 log 檔。內容應包含：
  - 作者或來源（例如：自動化、某位開發者）
  - 受影響之檔案路徑
  - 修改摘要（簡短）
  - 變更原因（簡述）
  - 必要時可附上修改前後的程式片段（簡略）
  - 修改後的完整程式碼（含行號） — 建議在 log 檔中加入修改後的完整檔案內容，並在每一行前標註行號，方便 code review 與追蹤差異。
- **規模說明**: 每個 log 檔以單一變更事件為單位（Small-to-Medium），適合記錄單一檔案或相關小範圍修改；若為大型重構或多模組重大改動，建議拆分為多個 log 檔或在 log 檔中標註關聯 ticket/PR 編號。
- **範例流程**: 在執行修改並確認無誤後，建立一個 timestamp log 檔，填寫上述欄位，然後將檔案加入版本控制（`git add && git commit`）。
- **自動化建議**: 可在 CI/CD 或 code-assistant 操作時自動產生此 log，並在 PR 描述中引用。範例自動化行為：
  1. 產生 log 檔（命名使用時間戳）
  2. 將 log 檔加入變更清單
  3. 在 PR 描述加入簡短連結與摘要

如果你想把此規範延伸成更嚴格的 CHANGELOG（例如遵循 Keep a Changelog 格式），我可以幫你把 README 改寫成該格式，或建立一個 `CHANGELOG.md` 的匯總檔來聚合 `log` 資料夾的內容。
